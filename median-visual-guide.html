Finding the middle number

Two sorted lists, one median — explained so simply a 12-year-old gets it

The basics

What is a median?

Imagine your class took a test. Everyone's score, sorted from low to high:

22
45
51
67
72
78
85
91
99

The median is just the middle number. Half the class scored below it, half scored above it. Here it's 72.

If there's an even number of scores, you take the average of the two middle ones.

The problem

Two classes, one middle

Now imagine two classes took the same test. Each class has their scores sorted already:

Class A
1
4
7
9
Class B
2
3
5
8
10
11

You want to find the median of ALL 10 scores combined.

The easy way? Merge them into one big sorted list and pick the middle. But that's slow for huge lists.

Can we find the middle without merging? Yes! Here's the trick...

The big idea

Build a wall between two halves

Forget sorting everything together. Instead, imagine putting a wall through both lines of students. Everyone on the left side has a smaller score, everyone on the right has a bigger score.

Can you split both lines so the left side = 5 students?
Class A
Class B
Move the wall: how many from Class A go to the left?
The key insight

Once you decide how many from A go left, B is automatic. If the left side needs 5 total and you take 2 from A, you must take 3 from B. That's it!

The check

How do you know the wall is in the right place?

Each class is already sorted within itself. So you only need to check across classes:

Check 1
A's biggest on the left must be ≤ B's smallest on the right
Check 2
B's biggest on the left must be ≤ A's smallest on the right

If both checks pass, the wall is in the right spot. If not, you know exactly which direction to move it:

A's left is too big?

A gave too many students to the left. Move the wall in A to the left (take fewer from A).

B's left is too big?

A gave too few students to the left. Move the wall in A to the right (take more from A).

Both pass?

The wall is perfect! The median is right there at the wall.

The speed trick

You don't try every position — use binary search!

Think of it like a guessing game. Your friend picks a number between 1 and 100. After each guess, they say "too high" or "too low".

You don't guess 1, 2, 3, 4... You guess 50 first, then 25 or 75, cutting the range in half each time. That's binary search.

Same thing here. We try the middle cut in A. If it's wrong, the cross-check tells us "too many" or "too few", and we cut the range in half.

Why always search the smaller class?

If Class A has 3 students and Class B has 1000, we only need to try positions in A (just 4 possibilities: 0, 1, 2, or 3). That's way faster than searching through B's 1001 positions!


Also, it prevents a crash. If we search the big array and pick a very small number, we might need more from B than B has. Searching the smaller array keeps us safe.

Story → Code

Now let's turn each idea into code

Every line below maps to something we just learned. No magic.

Code piece 1

Pick the smaller class to search

The story

You have Class A (8 students) and Class B (3 students). Searching 3 is faster and safer than searching 8. So if A is bigger, just swap them.

↓
if (A.length > B.length)
return findMedianSortedArrays(B, A); // swap!
In plain English

"If A is bigger than B, call the same function again but swap them. Now A is always the small one."

Code piece 2

How many go on the left side?

The story

10 total students → 5 on each side. 11 total → 6 on left, 5 on right. The left gets the extra one if it's odd.

↓
int total = A.length + B.length;
int leftHalfSize = (total + 1) / 2;
Why + 1?

For 10: (10+1)/2 = 5 (splits evenly). For 11: (11+1)/2 = 6 (left gets the extra). The +1 handles both odd and even with one formula. Integer division (no decimals) does the rest.

Code piece 3

The guessing game begins

The story

"How many from A go left?" Could be 0 (none) to 4 (all). Start guessing in the middle. If wrong, the check tells you which direction to go.

↓
int lo = 0, hi = A.length; // range: 0 to 4
while (lo <= hi) {
int fromA = (lo + hi) / 2; // guess the middle
int fromB = leftHalfSize - fromA; // B fills the rest
fromB is automatic!

If the left needs 5 total and you take 2 from A, then B must give 3. Always: fromB = leftHalfSize - fromA. You only choose one number, the other follows.

Code piece 4

Look at the students next to the wall

The story

After placing the wall, look at the 4 students standing right next to it — 2 on the left edge, 2 on the right edge. These are the only ones that matter for the cross-check.

↓
int leftA = fromA > 0 ? A[fromA - 1] : MIN_VALUE;
int rightA = fromA < A.length ? A[fromA] : MAX_VALUE;
int leftB = fromB > 0 ? B[fromB - 1] : MIN_VALUE;
int rightB = fromB < B.length ? B[fromB] : MAX_VALUE;
What's MIN_VALUE and MAX_VALUE?

What if A sends nobody to the left? There's no "last person on A's left side". So we use -∞ (the smallest possible number) as a stand-in. It will never cause a problem in the cross-check because everything is bigger than -∞.


Same idea: if A sends everyone to the left, there's nobody on A's right. We use +∞ (biggest possible number) as a stand-in.

Code piece 5

The cross-check

The story

Check: is A's biggest on the left ≤ B's smallest on the right? And is B's biggest on the left ≤ A's smallest on the right? If both yes — you found it!

↓
if (leftA <= rightB && leftB <= rightA) {
// FOUND IT!
} else if (leftA > rightB) {
hi = fromA - 1; // too many from A, guess lower
} else {
lo = fromA + 1; // too few from A, guess higher
}
Why only 2 checks?

Within each class, scores are already sorted. A's left is always ≤ A's right. Same for B. The only thing that could be wrong is across classes — and that's just 2 comparisons.

Code piece 6

Grab the median

The story

The wall is in the right place. The median is right there! If odd total, it's the biggest score on the left. If even, average the biggest on the left and smallest on the right.

↓
if (total % 2 == 1)
return Math.max(leftA, leftB); // biggest on the left
else
return (Math.max(leftA, leftB) + Math.min(rightA, rightB)) / 2.0;
